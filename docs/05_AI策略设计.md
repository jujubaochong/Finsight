# 05 AI 策略设计

> 本文说明 AI 在产品中的应用边界、模型选型、提示词策略、数据源方案与效果评估方法。

---

## AI 应用场景矩阵

| 场景 | 用AI? | 模型选择 | 原因 |
|------|:----:|---------|------|
| 财务数据展示（表格/图表） | 否 | — | 纯数据渲染，不需要 AI |
| 财务趋势分析（文字解读） | 是 | DeepSeek-V3 | 需要理解数据结构+生成中文分析文本 |
| 风险信号识别 | 混合 | 规则引擎 + DeepSeek-V3 | 确定性规则用代码，语义判断用 AI |
| 公告摘要 | 是 | DeepSeek-V3 | 长文本提炼，中文 NLP 核心场景 |
| 一句话总结 | 是 | DeepSeek-V3 | 综合多维度信息生成概述 |
| 报告章节生成 | 是 | DeepSeek-V3 + Claude | DeepSeek做批量章节，Claude做复杂推理章节 |
| 异动判断 | 否 | 规则引擎 | 规则定义清晰（"减持>1%→P1"），不需要AI |
| Embedding/知识检索 | 是 | bge-large-zh-v1.5 | 中文 SOTA，开源可自部署 |

## 模型选型

### 主模型：DeepSeek-V3

| 维度 | 评价 |
|------|------|
| 中文金融文本理解 | ★★★★★ 国内最强之一 |
| 长文本处理 | ★★★★★ 支持128K context |
| 成本 | $0.27/1M input tokens，约是 GPT-4o 的 1/10 |
| 私有化部署 | 支持，对金融客户数据敏感场景友好 |
| 速度 | 快，简单查询2-3秒，复杂分析15-30秒 |

### 辅助模型：Claude 4 Sonnet

仅在以下场景使用，用于 DeepSeek 表现不够好的深度推理：
- 跨文档矛盾识别（公告说"经营正常"，财报显示现金流枯竭 → 需要深度推理）
- 复杂财务逻辑分析（如商誉减值测试合理性评估）

### Embedding 模型：bge-large-zh-v1.5

- 用途：将历史研报、公告文档向量化，支持相似检索
- 选择理由：中文 MTEB 榜单前列，开源可自部署，不需要调 API

### 为什么不用单一模型

| 如果全用 DeepSeek | 深度推理不如 Claude |
|如果全用 Claude | 日常分析成本太高（$3 vs $0.27/1M tokens），一个完整研报可能消耗 $2-5 |
| 如果全用本地模型 | 部署成本高，中文金融理解不如商业模型 |

策略：**90% DeepSeek + 10% Claude 兜底**，成本控制在合理范围。

## 提示词策略

### 快速分析 System Prompt

```
你是一名经验丰富的A股分析师。你的任务是基于提供的股票数据，输出一份
简洁、客观的分析摘要。

== 角色 ==
- 你擅长从财务数据中发现值得关注的信息
- 你的分析风格是"客观、冷静、有数据支撑"
- 你会主动标注不确定的地方

== 输入数据 ==
你会收到以下结构化数据：
- 股票基本信息（代码、名称、行业、市值等）
- 近8个季度的核心财务指标
- 近30天公告标题和摘要
{数据内容由系统自动填充}

== 输出格式 ==
严格按照以下JSON格式输出，不要输出其他内容：

{
  "summary": "一句话概述这家公司当前的经营状况和值得关注的变化",
  "strengths": [
    "亮点1：引用具体数据",
    "亮点2",
    "亮点3"
  ],
  "risks": [
    "风险1：引用具体数据，说明为什么值得关注",
    "风险2",
    "风险3"
  ],
  "metrics_commentary": "对核心指标变化的综合解读，2-3句话"
}

== 严格约束 ==
1. 所有财务数据必须从提供的源数据中引用，绝对不能编造或自行计算
2. 引用的数字必须和源数据完全一致（包括小数点）
3. 不能给出买卖建议或股价预测
4. 不确定的判断标注"[需核实]"
5. 亮点和风险都要有数据支撑，不能是空泛的评价
6. 如果某个方面数据不足，标注"该维度数据不充分，无法分析"
```

### 公告摘要 System Prompt 要点

```
你是公告信息提炼助手。输入一份上市公司公告原文，输出3句话以内的核心
信息摘要。

规则：
- 第一句：公告类型（业绩预告/股东减持/资产重组/诉讼/其他）
- 第二句：核心数字或关键变化
- 第三句（可选）：对投资者的可能影响（客观陈述，不做价值判断）
- 如果公告内容为常规披露（如董事会决议通过一项常规议案），标注"常规公告"
- 输出格式：纯文本，不超过100字
```

### 为什么要链式调用

```
方案A（端到端）：给 AI 一堆原始数据 → 要求输出完整报告
  问题：Token 消耗巨大（全部数据一次性输入），中间环节黑盒，
        某章节出问题只能全部重来。

方案B（链式调用）：分步处理
  Step 1: 数据标准化（无AI，纯代码）→ 格式化财务数据
  Step 2: 并行分析（AI × 4）
    ├── 财务趋势解读
    ├── 风险信号识别
    ├── 公告摘要提炼
    └── 机构观点聚合（如有）
  Step 3: 报告组装（AI × 1）→ 将各模块结果组织成完整报告

  优势：
  - 每步 Token 可控（单次 2K-8K tokens）
  - 单步失败只重试该步
  - 可在 Step 2 和 Step 3 之间插入人工审核（V1规划）
  - 并行调用将总耗时控制在 30 秒内

采用方案B。
```

## 数据源方案

| 数据类型 | 数据源 | 获取方式 | 更新频率 | 成本 |
|---------|--------|---------|:------:|------|
| 股票基础信息 | AkShare | Python SDK 调用 | 每日 | 免费 |
| 财务报表 | AkShare | Python SDK 调用 | 每季度+每交易日更新 | 免费 |
| 公告列表+内容 | 巨潮资讯 / 东方财富 | 爬虫或 AkShare 封装 | 每30分钟 | 免费 |
| 行业分类 | 申万一级行业 | AkShare 行业接口 + 成分股映射 | 每周 | 免费 |
| 新闻舆情 | 暂不接入（MVP） | — | — | — |
| 券商研报 | 暂不接入（V1） | 东方财富研报板块 | 每日 | 可能有反爬 |

> **为什么 MVP 用 AkShare？** 免费、覆盖 A 股全量数据、Python SDK 开源。缺点是接口偶尔不稳定、且接口结构可能变动——处理方式：每个外部调用加 try-catch + 重试（指数退避）+ 结果缓存 + 真实数据不可用时降级到占位数据并标注，保证服务不中断。

## 效果评估方案

### 怎么评估 AI 分析质量

不是先上线再评估，而是开发阶段就评估：

**Step 1：准备测试集**

选 10 只覆盖不同行业（消费/制造/TMT/金融/医药各2只）、不同生命周期（成长期3只/成熟期5只/困境期2只）的股票。这保证了评估不是"挑好股票测"。

**Step 2：5维度人工打分**

| 评分维度 | 1分（差） | 3分（合格） | 5分（优秀） |
|---------|---------|-----------|----------|
| 数据准确性 | AI引用数字≥3处错误 | 数字正确但偶有格式问题 | 所有数字与源数据完全一致 |
| 分析深度 | "营收增长12%"（纯播报） | 说明了增长原因和可持续性 | 关联了多项指标，揭示了深层规律 |
| 可读性 | 术语堆砌，普通人看不懂 | 大体可读 | 逻辑清晰，外行也能理解大意 |
| 是否有幻觉 | 存在编造的数据或事实 | 无幻觉但有模糊表述 | 所有判断有依据，不确定性明确标注 |
| 是否有价值 | 看完跟没看一样 | 有一两处有用的信息 | 看完对股票有了新的认识 |

综合评分 = 5维度平均分。目标：≥ 3.5 分（合格线上）。

**Step 3：盲测对比**

找一份真实券商研报（买方的深度报告），在同一只股票上对比 FinSight AI 报告 vs 券商研报。找差距：
- AI 漏了什么？（通常是行业深度、管理层判断、实地调研信息——这些 AI 做不到）
- AI 多做了什么？（通常在风险信号枚举上比人类更全面）

目标：AI 报告在"数据全面性"上不输券商，在"分析深度"上差距可接受。

### 评估指标

| 指标 | 定义 | 如何测量 | 目标 |
|------|------|---------|:------:|
| 财务数据准确率 | AI引用的数字中，与源数据完全一致的占比 | 人工抽查100个引用点 | > 95% |
| 幻觉率 | AI编造的不存在信息 / 总生成内容 | 人工评测10份报告 | < 5% |
| 分析深度评分 | 上述5维度综合评分 | 3人独立打分取中位数 | ≥ 3.5/5 |
| 用户读完率 | 用户打开报告后滚动到>80%位置的比例 | 前端埋点（滚动深度） | > 40% |
| 报告采纳率 | 用户导出/分享报告的行为占比 | 导出按钮点击 / 报告打开 | > 25% |

### 评测节奏

| 阶段 | 方式 | 频率 |
|------|------|:--:|
| 开发中 | PM自己测（用熟悉的股票） | 每改一次 prompt 测一次 |
| 提测前 | 完整10只股票测试集打分 | 一次完整评测 |
| 上线后 | 用户行为数据 + NPS + 反馈 | 持续 |

## Badcase 处理

### 最可能出现的 3 个 badcase

**Badcase 1：AI 搞混了报告期**

表现：AI 说"公司 2025Q1 营收增长 30%"，实际上是环比增长而非同比增长。

根因：Prompt 中没有明确要求区分同比 vs 环比。

修复：System Prompt 强制要求"涉及增长率时必须标注比较基准（同比/环比/对比报告期）"。

**Badcase 2：某行业公司财务特征特殊，AI 通用分析框架不适用**

表现：分析银行股时，AI 拿"资产负债率"说事——但银行本身就是高杠杆经营。

根因：通用分析 prompt 不区分行业。

修复：System Prompt 中加入行业特殊规则：
- 银行/金融：不关注"资产负债率"，关注"不良贷款率、拨备覆盖率、净息差"
- 医药：关注"研发费用率、核心产品集中度"
- 消费：关注"经销商数量变化、预收账款"

**Badcase 3：公司刚好做过资产重组，历史数据不可比**

表现：AI 对比"今年收入 vs 去年收入"但没有考虑公司并表了新子公司。

根因：AI 不知道公司发生了重大资产重组。

修复：公告摘要中如果有"资产重组""重大资产购买"关键词，在分析报告中标注"该公司近期发生过重大资产重组，历史财务数据可比性可能受到影响"。
